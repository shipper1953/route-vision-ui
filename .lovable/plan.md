# Fix missing Easyship label after successful purchase

## Root cause

Easyship's `POST /2024-09/shipments/{id}/label` is **asynchronous**. It returns `202` (or a shipment object with `label.label_url` still null) and generates the label in the background. Our current `purchaseEasyshipLabel` function in `supabase/functions/purchase-label/index.ts` reads the immediate response and returns `label_url: undefined` and `tracking_number: undefined`.

The shipment is then saved to the DB with `label_url = null`, the wallet is charged, and the order is marked shipped. The UI's `ShippingLabelDialog` falls back to its "No label available" empty state — which is exactly what your screenshots show.

Sandbox mock couriers can also legitimately *never* produce a real PDF, so we also need a graceful fallback path.

## Plan

### 1. Poll for label readiness inside `purchase-label` (primary fix)

In `supabase/functions/purchase-label/index.ts`, after the `POST /shipments/{id}/label` call, if `label_url` or `tracking_number` are missing, poll `GET /2024-09/shipments/{id}` every 2 seconds, up to 8 attempts (~16s total — well under the edge-function timeout).

- Stop early as soon as both `label_url` and `tracking_number` are populated.
- If Easyship reports `label_state === 'failed'`, throw with the courier's error message so the user sees a real error instead of a silent empty dialog.
- If the polling budget elapses without a label, return `label_pending: true` alongside the partial data. The shipment still saves (so wallet/order state is consistent) but the UI knows to show a "label generating" state with a refresh button.

### 2. Add `refresh-easyship-label` edge function (fallback)

New function `supabase/functions/refresh-easyship-label/index.ts`:

- Auth: validate caller via JWT and confirm they belong to the same `company_id` as the shipment row.
- Input: `{ shipmentId: number }` (our DB id).
- Looks up the shipment, calls `GET /2024-09/shipments/{easypost_id}` against Easyship.
- If `label_url` / `tracking_number` are now present, updates the `shipments` row (`label_url`, `tracking_number`, `tracking_url`) and returns the fresh values.
- Otherwise returns `{ status: 'pending' }` so the UI can keep polling or let the user retry later.

### 3. Update `ShippingLabelDialog.tsx` to handle the pending state

- When `labelUrl` is missing but a `shipmentId` is present, render a "Label is still being generated by the carrier" panel with:
  - Tracking number (if we have it).
  - A **Refresh label** button that calls `refresh-easyship-label` and updates local state when the URL becomes available.
  - A short note that sandbox/mock couriers may never produce a PDF.
- Keep the existing PDF preview / download / print UI for the case when the label *is* available — no regressions for EasyPost/Shippo.
- Pass the new `label_pending` flag through from `CreateShipment.tsx` so the dialog can distinguish "still generating" from "no label at all".

## Files

- `supabase/functions/purchase-label/index.ts` — add polling loop in `purchaseEasyshipLabel`, expose `label_pending`.
- `supabase/functions/refresh-easyship-label/index.ts` — **new**, manual re-poll + DB update.
- `src/components/shipment/ShippingLabelDialog.tsx` — pending-state UI + refresh button.
- `src/pages/CreateShipment.tsx` — forward `label_pending` to the dialog.

## Out of scope / explicit non-goals

- We will **not** restructure the call into a 202-then-client-poll architecture. The 16s in-function poll covers the common case; the manual refresh button covers the long-tail/sandbox case. This is simpler and avoids a second round of edge-function plumbing.
- No DB schema changes — `shipments.label_url` already nullable.

## Acceptance

- Real Easyship rate purchased in sandbox → polling fills in `label_url` within ~2–6s and dialog shows the PDF preview / Print / Download as usual.
- Sandbox mock courier that never returns a PDF → dialog shows "Label still generating" with a Refresh button; clicking it calls `refresh-easyship-label` and either pulls in the label or stays pending without errors.
- EasyPost and Shippo flows unchanged.
